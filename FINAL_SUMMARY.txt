ANFORDERUNGEN ✅ UMGESETZT
===========================

1. ✅ LIGA-STRUKTUR
   • League-Modell erweitert um Division (1, 2, 3) und DivisionLabel
   • 7 Ligen möglich: 1x 1. Liga, 2x 2. Liga, 4x 3. Liga
   • Jede Liga mit 12 Slots (Teams)

2. ✅ INITIALISIERUNG
   • 7 Ligen werden automatisch beim Server-Start erstellt
   • 84 CPU-Teams (12 pro Liga) werden generiert
   • Jedes Team erhält:
     - 18 Spieler (mit Positionen und Attributen)
     - 3 Lineups (4-4-2, 4-3-3, 3-5-2)
     - 30 Stadionteile
   • Schedules werden für jede Liga erstellt
   • ApplicationStartupListener orchestriert alles

3. ✅ USER-REGISTRIERUNG
   • Neue Registrierungs-Route mit Liga-Wahl
   • User wählt Liga bei der Anmeldung
   • Zufälliger CPU-Team-Slot wird mit User-Team ersetzt
   • Altes CPU-Team wird gelöscht
   • User erhält leagueId in seiner Entity

4. ✅ SPIELPLAN MIT LIGA-ANSICHT
   • Standard: User sieht eigene Liga
   • Dropdown zum Wechsel zwischen Ligen
   • Jede Liga hat eigene Tabelle und Spielplan
   • API-Endpoints für Liga-Wechsel

AUTHENTIFIZIERUNG
==================

GET /api/auth/leagues
  Beschreibung: Alle verfügbaren Ligen mit Metadaten
  Response: Array<LeagueInfoDTO>
  Beispiel:
    [
      { id: 1, name: "1. Liga", division: 1, divisionLabel: "1. Liga", 
        filledSlots: 12, totalSlots: 12 },
      { id: 2, name: "2. Liga A", division: 2, divisionLabel: "2. Liga A",
        filledSlots: 11, totalSlots: 12 },
      ...
    ]

POST /api/auth/register-with-league
  Beschreibung: Registrierung mit Liga-Wahl
  Body: { username, password, teamName, leagueId }
  Response: { token, teamId }
  Beispiel:
    Request:  { username: "spieler1", password: "pw123", teamName: "FC Test", leagueId: 1 }
    Response: { token: "abc-123", teamId: 42 }

SPIELPLAN
=========

GET /api/v2/schedule/leagues
  Beschreibung: Alle Ligen (wie /api/auth/leagues)
  Response: Array<LeagueInfoDTO>

GET /api/v2/schedule/standings
  Beschreibung: Tabelle der User-Liga (Standard)
  Response: Array<LeagueStandingsDTO>

GET /api/v2/schedule/standings/league/{leagueId}
  Beschreibung: Tabelle einer spezifischen Liga
  Response: Array<LeagueStandingsDTO>
  Beispiel:
    GET /api/v2/schedule/standings/league/1
    [
      { teamId: 5, teamName: "Bayern", position: 1, played: 10, won: 8, 
        drawn: 1, lost: 1, goalsFor: 28, goalsAgainst: 8, points: 25, strength: 445 },
      ...
    ]

ADMIN
=====

POST /api/admin/initialize-leagues
  Beschreibung: Manuell Ligen initialisieren (normalerweise nicht nötig)
  Response: { success: true, message: "..." }

BEIM SERVER-START:
  1. Spring Boot startet ManagerApplication
  2. ApplicationStartupListener triggered on ApplicationReadyEvent
  3. Ruft RepositoryService.initializeLigues() auf
  4. Erstellt 7 Ligen mit 84 CPU-Teams
  5. Jedes Team erhält 18 Spieler und Lineups
  6. Schedules werden generiert
  7. Server bereit ✓

BEIM USER-REGISTRIEREN:
  1. Frontend zeigt GET /api/auth/leagues Dropdown
  2. User wählt Liga (z.B. "3. Liga C")
  3. Frontend sendet POST /api/auth/register-with-league
  4. Backend erstellt neues User-Team
  5. Backend findet Liga "3. Liga C"
  6. Backend findet zufälligen CPU-Slot (z.B. Slot 7)
  7. Backend ersetzt CPU-Team durch User-Team
  8. Backend löscht altes CPU-Team
  9. Backend erstellt User mit leagueId
  10. User wird eingeloggt mit Token
  11. Frontend speichert Token + TeamId

WENN USER SPIELPLAN ÖFFNET:
  1. Frontend lädt GET /api/v2/schedule/standings (User-Liga)
  2. Zeigt Tabelle mit allen Teams der User-Liga
  3. Frontend zeigt Liga-Dropdown (GET /api/v2/schedule/leagues)
  4. User kann Liga wechseln
  5. Bei Wechsel: GET /api/v2/schedule/standings/league/{newLeagueId}
  6. Neue Tabelle wird angezeigt
